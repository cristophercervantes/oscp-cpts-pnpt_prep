## 1. What is Cryptography?

At its core, cryptography is about **securing communication when someone is actively trying to break it**. Not just keeping secrets from a nosy neighbor — we're assuming the adversary is genuinely dangerous.

The classic setup involves three parties:

- **Alice** – the sender
- **Bob** – the receiver
- **Eve** – the adversary sitting in the middle

Eve isn't just passively listening. She can **read, modify, inject, and delete** data. The key mindset here: assume the worst. The adversary is malicious, powerful, and unpredictable.

## 2. Fundamental Goals

These are the things cryptography is actually trying to achieve:

1. **Confidentiality** — Only the intended recipients can read the data. Nobody else.
2. **Data Integrity** — The data hasn't been tampered with on the way. What Bob receives is exactly what Alice sent.
3. **Data Origin Authentication** — Bob can verify that the message actually came from Alice, not some impersonator.
4. **Non-repudiation** — Alice can't later deny that she sent something. Think of it like a digital signature on a contract.

Worth noting: goals 2 and 3 together fall under what's broadly called _authentication_.

## 3. Where Cryptography Actually Shows Up

It's easy to think of crypto as abstract theory, but it's running underneath almost everything:

- **Secure browsing (TLS 1.3)** — Every time you see `https://` in your browser, TLS is doing the work.
- **Automatic software updates** — When your OS downloads and installs an update, signatures verify it's the real deal (e.g. Microsoft signing Windows updates).
- **Cell phone calls (GSM)** — The wireless link between your phone and the tower is encrypted.
- **Bluetooth** — Your phone to your headphones? Crypto is there too.
- **Secure messaging (Signal, WhatsApp)** — End-to-end encryption so even the app provider can't read your messages.
- **Cloud computing (AWS, Dropbox, Netflix)** — Data in transit and at rest relies on crypto. And the "adversary" here might include foreign governments or intelligence agencies — hence why this is taken seriously.

## 4. Usage: Secure Web Transactions (TLS)

This section is worth understanding in detail because TLS is the most visible real-world crypto deployment.

### The Problem

Alice (your browser) wants to talk securely to Bob (e.g. Gmail's servers). The internet between them is completely untrusted — Eve could be anywhere.

### Step 1 — Symmetric-key crypto

If Alice and Bob already share a secret key `k`, they can encrypt with **AES** and authenticate with **HMAC**. Simple enough. But the hard question is: **how do they get that shared key in the first place?** They've never met before.

### Step 2 — Public-key crypto to the rescue

Instead of a shared secret, each party has a **key pair**: a public key (anyone can see it) and a private key (kept secret). Alice can encrypt something with Bob's _public_ key, and only Bob can decrypt it with his _private_ key.

So Alice picks a random session key `k`, encrypts it with Bob's RSA public key, sends it over. Bob decrypts it. Now they both have `k` and can communicate securely.

But wait — **how does Alice know she has Bob's real public key?** Eve could have swapped it with her own!

### Step 3 — Certificates and Certificate Authorities

This is where **Certification Authorities (CAs)** come in. A CA (like DigiCert) is a trusted third party that vouches for a server's identity. Here's how it works:

- Bob (the server) gets his public key _signed_ by the CA using the CA's private signing key. This signed bundle is called a **certificate**.
- When Alice visits a website, the server sends her this certificate.
- Alice's browser verifies the CA's signature using the CA's public key — which is **pre-installed in the browser** by the browser makers.
- If the signature checks out, Alice knows Bob's public key is genuine.

### The Full TLS Flow

1. Alice visits `https://gmail.com`. Gmail's server sends its **certificate**.
2. Alice's browser verifies the certificate using the CA's embedded public key.
3. Alice picks a random session key `k`, encrypts it with Gmail's RSA public key, sends it over.
4. Gmail decrypts it to get `k`.
5. From now on, all communication in that session is encrypted with AES and authenticated with HMAC using `k`.
6. The **padlock icon** in your browser confirms this is happening.

## 5. Potential Vulnerabilities of TLS

Even with all of this, things can go wrong. Here are the realistic attack surfaces:

1. **Weak cryptography** — If AES, HMAC, or RSA have flaws, the whole thing falls apart.
2. **Quantum computers** — Shor's algorithm can break RSA. This is a looming problem for the future.
3. **Bad random number generation** — Crypto relies heavily on randomness. Poor RNG is a silent killer.
4. **Fraudulent certificates** — In 2001, Verisign accidentally issued certificates to someone pretending to be a Microsoft employee. This is a real and recurring problem.
5. **Software bugs** — Both accidental (Heartbleed) and intentional (supply chain attacks).
6. **Phishing** — You can have perfect crypto and still get tricked into talking to the wrong server.
7. **TLS only protects data in transit** — Once data sits on the server, TLS is done. Server-side breaches are a different problem entirely.

## 6. Cryptography vs. Cybersecurity

This is an important distinction the course draws clearly.

**Cybersecurity** is the broader field — it covers:

- _Computer security_: secure OS design, antivirus, auditing, risk management
- _Network security_: firewalls, intrusion detection, DoS protection, protocol design
- _Software security_: buffer overflows, compilers, DRM, code obfuscation

**Cryptography** is a small piece of this — but it's indispensable. The course uses a good analogy: **security is a chain**. The weakest link is where attackers focus. Crypto usually isn't the weakest link... but when it fails, the consequences are catastrophic. There's no patching a broken cipher after the fact.

One of the XKCD comics in the slides makes this point nicely: a crypto nerd imagines spending millions cracking 4096-bit RSA, while in reality, the attacker just... threatens someone until they give up the password. Humans are often the weakest link, not the math.
