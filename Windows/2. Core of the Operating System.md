## Windows Operating System Structure

In Windows, the directory structure starts at the **Root Directory**, typically designated as the `C:\` drive (the boot partition). This is where the core operating system and its supporting files reside.

### 1. System Directories

The boot partition follows a standardized structure to manage programs, user data, and system configurations:

|**Directory**|**Function**|
|---|---|
|**Windows**|The heart of the OS; contains the majority of files required for Windows to run.|
|**System32 / SysWOW64**|Contains essential **DLLs** (Dynamic Link Libraries) and the Windows API. System32 holds 64-bit files, while SysWOW64 holds 32-bit files on 64-bit systems.|
|**Program Files**|Installation path for 64-bit applications.|
|**Program Files (x86)**|Installation path for 32-bit applications (on 64-bit systems).|
|**ProgramData**|A hidden folder containing data shared across all users that is essential for specific applications to function.|
|**Users**|Contains individual user profiles. Includes the **Default** template for new users and the **Public** folder for local file sharing.|
|**AppData**|A hidden subfolder within each user profile (`Users\<user>\AppData`) that stores per-user settings.|

### 2. The AppData Subfolders

The `AppData` directory is divided into three specific areas based on how the data is handled:

- **Roaming:** Contains machine-independent data (like custom dictionaries) that follows a user profile across different computers on a network.
    
- **Local:** Data specific to the local machine; it is never synchronized across a network.
    
- **LocalLow:** Similar to Local but with a lower integrity level, often used by web browsers in "Protected Mode" to limit potential damage from malicious scripts.
    
### 3. Exploring Directories via Command Line

Windows provides built-in tools to navigate and visualize the file system without a Graphical User Interface (GUI).

#### The `dir` Command

Used to list files and subdirectories.

- **`dir /a`**: Displays all files, including hidden and system files (like `pagefile.sys` or `ntldr`).
    
- **`dir c:\windows`**: Lists the contents of the Windows directory.
    

#### The `tree` Utility

Provides a graphical "tree" representation of the directory structure, which is helpful for understanding complex application folders.

- **`tree [path]`**: Displays the folder structure.
    
- **`tree [path] /f`**: Displays the folder structure **and** the individual files within them.
    
- **`tree c:\ /f | more`**: Walks through every file on the C drive, pausing after each screen of text.
    
### 4. Critical System Files

When listing the root directory with `dir /a`, you will often see several large, protected system files:

- **pagefile.sys**: Used as Virtual Memory when physical RAM is full.
    
- **hiberfil.sys**: Stores the state of the OS when the computer enters "Hibernate" mode.
    
- **WinSxS**: The "Windows Component Store," which keeps copies of all updates and service packs to allow the system to roll back if an update fails.

## Windows File Systems and NTFS Permissions

Windows has evolved through several file systems, but modern operations primarily revolve around **FAT32**, **exFAT**, and **NTFS**. Understanding these is crucial for managing data storage and system security.

### 1. FAT32 vs. NTFS

While FAT32 is excellent for cross-platform compatibility (working across Windows, Mac, and Linux), NTFS is the standard for modern Windows installations due to its advanced features.

|**Feature**|**FAT32**|**NTFS**|
|---|---|---|
|**Max File Size**|4 GB|16 EB (Exabytes)|
|**Security**|None (No file permissions)|**Granular Permissions** (ACLs)|
|**Reliability**|Basic|**Journaling** (Logs changes to recover from crashes)|
|**Compression**|Not built-in|Supported natively|
|**Compatibility**|High (Mobile, Cameras, TVs)|Limited to Windows and newer macOS (Read-only usually)|

### 2. NTFS Permission Types

NTFS allows administrators to define exactly what a user can do with a file or folder. These are often **inherited** from parent folders to save time.

- **Full Control:** The "God mode" of permissions—read, write, change, and delete.
    
- **Modify:** Similar to Full Control but cannot change permissions or take ownership.
    
- **Read & Execute:** Allows viewing files and running applications.
    
- **List Folder Contents:** Allows seeing what’s inside a folder (applies to folders only).
    
- **Traverse Folder:** A unique permission that allows a user to pass through a folder they can't "read" to reach a subfolder or file they _do_ have access to.

### 3. Managing Permissions with `icacls`

While you can use the File Explorer GUI (Security tab), the command-line utility **`icacls`** (Integrity Control Access Control List) offers much more power and speed for auditing and automation.

#### Viewing Permissions

Running `icacls C:\Windows` will show a list of users and their access levels in a coded format:

- **F:** Full access
    
- **M:** Modify access
    
- **RX:** Read and Execute
    
- **R / W:** Read-only / Write-only
    

#### Inheritance Flags

You will often see parentheses like `(OI)` or `(CI)`. These tell you how permissions flow down to sub-items:

- **(OI) - Object Inherit:** Files below this folder will inherit this permission.
    
- **(CI) - Container Inherit:** Folders below this folder will inherit this permission.
    
- **(IO) - Inherit Only:** The permission doesn't apply to the current folder, only to its children.
    

#### Modifying Permissions

As a penetration tester or administrator, you can grant or revoke access quickly:

- **Granting Access:** `icacls C:\Users /grant joe:f` (Grants user 'joe' Full access to the folder).
    
- **Removing Access:** `icacls C:\Users /remove joe` (Removes all permissions for 'joe').
### 4. Security Implications

In a Red Team context, misconfigured NTFS permissions are a goldmine.

- **Weak Permissions:** If a low-privileged user has `(W)` or `(M)` access to a folder containing a service executable (like in `C:\Program Files`), they can replace the legitimate file with a malicious one to achieve **Privilege Escalation**.
    
- **Sensitive Data:** Checking `icacls` on backup directories or user folders often reveals data that was intended to be private but suffered from "inheritance" issues.


## NTFS vs. Share Permissions

When a folder is shared over a network (using the **SMB** protocol), two different sets of permissions act as a "double-gate" for security. Understanding the interaction between **Share Permissions** and **NTFS Permissions** is critical for both securing an environment and identifying paths for lateral movement during a pentest.

### 1. Share Permissions

These are the **first gate** a user hits when accessing a resource over the network (e.g., via `\\Server\Share`). They _only_ apply to network access and are less granular than NTFS.

- **Full Control:** Includes all "Change" and "Read" rights, plus the ability to modify permissions.
    
- **Change:** Allows users to read, edit, delete, and add files/folders.
    
- **Read:** Allows users to view the contents and names of files/folders.
    

### 2. NTFS Permissions

These are the **second gate** and apply to everyone—whether they are accessing the file over the network or logged in locally (via RDP or physical console).

- **Basic Permissions:** Common settings like _Read & Execute_, _Modify_, and _List Folder Contents_.
    
- **Special/Advanced Permissions:** Extremely granular controls, such as:
    
    - **Traverse Folder:** Allows reaching a subfolder even if the parent is restricted.
        
    - **Write Attributes:** Allows changing "Read-only" or "Hidden" flags without changing the file content.
        
    - **Take Ownership:** Allows a user to become the owner, granting them power to change any other permission.
        
### 3. The "Most Restrictive" Rule

When a user accesses a file over the network, Windows evaluates both sets of permissions and applies the **most restrictive** result.

> **Example:**
> 
> - **Share Permission:** Full Control
>     
> - **NTFS Permission:** Read-Only
>     
> - **Result:** The user can only **Read**.
>     

> **Example:**
> 
> - **Share Permission:** Read-Only
>     
> - **NTFS Permission:** Full Control
>     
> - **Result:** The user can only **Read**.
>     

### 4. Security & Malware Spread

Network shares are a primary vector for malware (like Ransomware) to spread across an organization.

- **Lenient Permissions:** If a share is set to "Everyone: Full Control," a single infected machine can encrypt files across the entire network.
    
- **EternalBlue (MS17-010):** This infamous exploit targets the SMBv1 protocol, allowing attackers to execute code and move laterally between systems. Disabling SMBv1 and properly configuring permissions are vital defense steps.

## Creating and Connecting to a Network Share

Establishing a network share involves configuring both the software service (SMB) and the security layers (Firewall, Share Permissions, and NTFS Permissions).

### 1. Creating the Share (Windows GUI)

To share a folder, you use the **Advanced Sharing** properties. This allows you to define a **Share Name** (which defaults to the folder name) and set a limit on simultaneous connections—a critical performance setting in enterprise environments.

### 2. The Two-Layer Permission Model

Every shared resource in Windows is governed by two distinct Access Control Lists (ACLs):

1. **Share Permissions (SMB):** Managed in the _Sharing_ tab. These determine access levels (**Read, Change, Full Control**) specifically for users connecting over the network.
    
2. **NTFS Permissions:** Managed in the _Security_ tab. These offer granular control (**Modify, Read & Execute, etc.**) and apply to both network and local users.
    

**The "Most Restrictive" Rule:** When accessing a share remotely, Windows calculates the "effective permission" by taking the most restrictive setting between the two lists. For example, if Share Permissions allow "Full Control" but NTFS only allows "Read," the user can only **Read**.

### 3. Connecting from Linux (cURL/smbclient)

As a penetration tester, you often use the **`smbclient`** utility to interact with Windows shares from a Linux environment like Kali or Pwnbox.

- **List Shares:** Identify which folders are available on the target.
    ```
    smbclient -L <TARGET_IP> -U <username>
    ```
    
- **Connect Interactively:** Access the share in an FTP-like shell.
    ```
    smbclient '//<TARGET_IP>/Company Data' -U <username>
    ```
    
### 4. Mounting the Share

To treat the Windows share like a local folder on your Linux desktop, you can **mount** it using the Common Internet File System (CIFS).
```
sudo mount -t cifs -o username=htb-student,password=Academy_WinFun! //<TARGET_IP>/"Company Data" /home/user/Desktop/share_mount
```

- **Note:** If this fails, ensure `cifs-utils` is installed (`sudo apt install cifs-utils`).
    
### 5. Troubleshooting: The Firewall Barrier

If your credentials are correct but the connection times out, the **Windows Defender Firewall** is likely blocking Port **445 (SMB)**.

- **Firewall Profiles:** Windows uses different rules for **Public, Private, and Domain** networks.
    
- **Predefined Rules:** Instead of turning off the firewall, you should enable the "File and Printer Sharing" inbound rules.
    
### 6. Monitoring and Auditing

Windows provides several tools to see who is connected to your shares:

- **`net share` (CLI):** Quickly lists all active shares, including hidden administrative shares like `C$` and `ADMIN$`.
    
- **Computer Management (GUI):** Under _Shared Folders_, you can see active **Sessions** (who is connected) and **Open Files** (what they are looking at).
    
- **Event Viewer:** Provides "journal entries" for every login and file access, essential for forensic investigations after a breach.
