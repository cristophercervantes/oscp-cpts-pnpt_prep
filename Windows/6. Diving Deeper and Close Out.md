## Desktop Experience vs. Server Core

### What's the Difference?

When installing Windows Server, you choose between two modes — and as of **Windows Server 2019**, this choice is permanent. You cannot switch between them after installation.

| Feature            | Desktop Experience | Server Core            |
| ------------------ | ------------------ | ---------------------- |
| Has a GUI?         | Full GUI           | Command-line only      |
| Resource usage     | Higher             | Lower                  |
| Attack surface     | Larger             | Smaller                |
| Ease of management | Easier             | Steeper learning curve |
| First available    | Long-standing      | Windows Server 2008    |

### Server Core — The Lean Option

Server Core strips Windows Server down to just what's needed to run server workloads. No full GUI, no bloat.

**Why choose it?**

- Lower disk and memory usage
- Fewer components = fewer things to patch and exploit
- Smaller attack surface

**How do you manage it?**

- Command Prompt / PowerShell
- Remote management via **MMC** or **RSAT**
- Initial setup via **Sconfig** — a text-based menu tool (runs as a VBScript) for tasks like networking, Windows updates, account management, and remote management config

### What Still Works on Server Core

Even without a full GUI, some graphical tools are still available:

**Built-in:**

- Registry Editor (`regedit`)
- Notepad
- Task Manager
- System Information
- Windows Installer
- PowerShell

**Sysinternals tools supported:**

- Active Directory Explorer
- Process Explorer
- Process Monitor
- TCPView

| --------- | ----------------------------------------------------------------------------------------------- |

## 1. Windows Security 

Windows has a large attack surface due to many built-in features, applications, and layers of settings. Even fully patched systems can be misconfigured. Microsoft continues to improve security with tools admins can use to harden systems, detect intrusions, and block misuse.

> **Windows applies security principles to users, computers, threads, and processes to minimize unauthorized access.

## 2. Security Identifier (SID)

Every security principal (user, group, computer) on a Windows system gets a unique SID automatically generated by the system. SIDs are stored in the security database and added to access tokens.

### SID Format

```
S-1-5-21-674899381-4069889467-2080702030-1002
(SID)-(revision)-(identifier-authority)-(subauthority1)-(subauthority2)-(RID)
```

### SID Components

|Part|Value|Meaning|
|---|---|---|
|S|S|Identifies string as a SID|
|Revision Level|1|Always 1 — never changed|
|Identifier Authority|5|48-bit — identifies who created the SID (5 = Microsoft)|
|Subauthority 1|21|Identifies user relation to the authority|
|Subauthority 2|674899381-...|Identifies the specific computer or domain|
|RID|1002|Relative ID — distinguishes user type (admin, guest, normal)|

> **REMEMBER:** Even two users with identical names have different SIDs. Windows uses SIDs — not names — to track permissions.

## 3. SAM, ACE, and ACL

### Security Accounts Manager (SAM)

SAM grants rights to a network to execute specific processes. It stores user account information and password hashes on the local machine.

### Access Control Lists (ACL) and Entries (ACE)

- **ACL** — a list attached to a resource (file, process) that defines who can access it
- **ACE** — a single entry inside an ACL defining one rule (e.g. "Bob can read this file")
- **DACL** (Discretionary ACL) — controls who has access to an object
- **SACL** (System ACL) — controls what access gets logged/audited

### Access Tokens

Every process/thread started by a user carries an access token. This token contains the user's SID plus group memberships and privilege levels. The Local Security Authority (LSA) validates these tokens.

> **ANALOGY:** ACL = guest list at a party. DACL decides who gets in. SACL records who tried to enter. Each ACE is one name on the list.

## 4. User Account Control (UAC)

UAC prevents malware from running or manipulating processes without user consent. It uses Admin Approval Mode to interrupt unauthorized installs and system-wide changes.

### How UAC Works

1. User attempts a privileged operation (e.g. installing software)
2. Windows calls ShellExecute → CreateProcess
3. Application Information Service evaluates the request
4. Checks: Is Secure Desktop enabled? Is app Windows-signed? What is the UAC slider level?
5. Consent prompt shown — user must confirm or enter admin password
6. Without confirmation: execution is denied

### UAC Slider Levels

|Level|Behavior|
|---|---|
|Always notify|Prompts for ALL changes (most secure)|
|Notify app changes (default)|Prompts only when apps make changes|
|Notify without secure desktop|Same but prompt not isolated|
|Never notify|UAC disabled (least secure)|

> **SECURITY NOTE:** UAC can be bypassed. It is not a security boundary — it is a convenience feature. Many UAC bypass techniques exist for privilege escalation.

## 5. The Windows Registry

A hierarchical database storing low-level settings for Windows and applications. Divided into computer-specific and user-specific data. Open with `regedit` from the command line or Start menu.

### Root Keys (Hives)

|Root Key|Short|Contains|
|---|---|---|
|HKEY_LOCAL_MACHINE|HKLM|All local system settings (SAM, SECURITY, SYSTEM, SOFTWARE, HARDWARE, BCD)|
|HKEY_CURRENT_USER|HKCU|Settings for the logged-in user (stored in NTUSER.DAT)|
|HKEY_CLASSES_ROOT|HKCR|File type and protocol associations|
|HKEY_USERS|HKU|Settings for all user profiles on the machine|
|HKEY_CURRENT_CONFIG|HKCC|Current hardware profile info|

### Common Value Types

|Type|Stores|
|---|---|
|REG_SZ|Plain text string (most common — file paths, app names)|
|REG_DWORD|32-bit number (on/off flags: 0 = off, 1 = on)|
|REG_QWORD|64-bit number|
|REG_BINARY|Raw binary data|
|REG_MULTI_SZ|List of strings (null-separated)|
|REG_EXPAND_SZ|String with environment variables (e.g. %PATH%)|

### Registry Hive Files on Disk

|Location|Contains|
|---|---|
|`C:\Windows\System32\Config\`|SYSTEM, SOFTWARE, SAM, SECURITY, DEFAULT, DRIVERS|
|`C:\Users\<name>\NTUSER.DAT`|Per-user HKCU hive|

### Run and RunOnce Keys 

These registry keys tell Windows to automatically run programs at startup or login. Attackers use them for persistence.

```
HKLM\Software\Microsoft\Windows\CurrentVersion\Run
HKCU\Software\Microsoft\Windows\CurrentVersion\Run
HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnce
HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce
```

```powershell
# Check startup programs
reg query HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run
```

> **ATTACK VECTOR:** Malware drops its .exe path into a Run key to survive reboots. Always check Run keys when investigating a compromised machine.

## 6. Application Whitelisting & AppLocker

### Whitelist vs Blacklist

|Blacklist|Whitelist (AppLocker)|
|---|---|
|Block known bad apps|Allow only approved apps|
|Everything else runs freely|Everything else is blocked by default|
|Reactive — updated as new malware appears|Proactive — unknown threats auto-blocked|
|Higher maintenance over time|Lower ongoing maintenance|

> Whitelisting follows a zero-trust model — all software is considered bad unless specifically approved. Recommended by NIST for high-security environments.

### AppLocker

Microsoft's built-in application whitelisting solution, introduced in Windows 7. Gives granular control over executables, scripts, Windows Installer files, DLLs, and packaged apps.

#### Rule Types

|Rule Type|Based On|Notes|
|---|---|---|
|Publisher|Digital signature / certificate|Most flexible — survives app updates. Recommended.|
|Path|File or folder location|Simple but risky — attacker can place malware in allowed path|
|Hash|Exact file fingerprint|Most strict — breaks on every update|

#### What AppLocker Controls

- Executables (`.exe`, `.com`)
- Scripts (`.ps1`, `.bat`, `.vbs`, `.js`)
- Windows Installer files (`.msi`, `.msp`)
- DLL files (`.dll`, `.ocx`)
- Packaged apps (Windows Store)

#### Deployment Approach

1. Deploy in **audit mode** first — logs what would be blocked without actually blocking
2. Review logs to find missed legitimate apps
3. Add missing apps to the whitelist
4. Switch to **enforcement mode**

> **REMEMBER:** Rules can be applied to security groups or individual users. AppLocker is configured via Local Group Policy Editor.

## 7. Local Group Policy

Allows admins to set, configure, and enforce a wide variety of system settings. In a domain environment, GPOs are pushed from a Domain Controller. On standalone machines, Local Group Policy applies.

```
Open: Start → type gpedit.msc   (not available on Windows Home)
```

### Two Main Sections

|Section|When Applied|Affects|
|---|---|---|
|Computer Configuration|At boot — before anyone logs in|The machine itself, all users|
|User Configuration|At login — per user session|The logged-in user only|

### Each Section Contains

- **Software Settings** — deploy/remove software
- **Windows Settings** — security policies, scripts, password rules
- **Administrative Templates** — hundreds of Registry-based settings

### Setting States

|State|Meaning|
|---|---|
|Not Configured|Group Policy silent — Windows uses its own default|
|Enabled|Policy is actively enforced|
|Disabled|Policy explicitly turned off — overrides Windows default|

### Useful Security Settings You Can Configure

- Password minimum length, complexity, and expiry
- Account lockout after X failed login attempts
- Disable command prompt / PowerShell for standard users
- Prevent access to Registry Editor
- Enable Credential Guard (Virtualization Based Security)
- Control Windows Update behavior
- Configure AppLocker rules
- Enable audit logging

### Credential Guard

Enabled via Group Policy: **Turn On Virtualization Based Security**. Protects against credential theft by moving the LSA (Local Security Authority) process into a hardware-isolated virtual environment. Even admin-level attackers cannot reach into it to steal login tokens.

>  Computer Configuration always wins over User Configuration if they conflict. System-level rules override personal preferences.

## 8. Windows Defender Antivirus

Built-in antivirus included free with Windows. Originally released as a downloadable anti-spyware tool for Windows XP/Server 2003. Became part of the OS with Windows Vista/Server 2008. Renamed Windows Defender Antivirus with Windows 10 Creators Update.

###  Features

|Feature|What It Does|
|---|---|
|Real-time protection|Scans every file as it is opened, copied, or executed|
|Cloud-delivered protection|Sends unknown files to Microsoft cloud for analysis; locks file until result returns|
|Tamper Protection|Prevents security settings being changed via Registry, PowerShell, or Group Policy|
|Controlled folder access|Blocks unauthorized apps from modifying protected folders — ransomware protection|
|Exclusion list|Lets admins mark trusted folders/files that Defender will not scan|
|Automatic sample submission|Auto-uploads suspicious files to Microsoft for threat intelligence|
|Behavior monitor|Watches running processes for suspicious behavior patterns|

### Checking Defender Status (PowerShell)

```powershell
Get-MpComputerStatus | findstr "True"
```

|Property|Meaning if True|
|---|---|
|AntivirusEnabled|Virus scanning is active|
|RealTimeProtectionEnabled|On-access scanning is running|
|BehaviorMonitorEnabled|Process behavior monitoring is active|
|IsTamperProtected|Settings are locked against tampering|
|NISEnabled|Network Inspection System is active|
|OnAccessProtectionEnabled|Files are scanned when accessed|

### Limitations

- Will detect payloads from common frameworks like Metasploit or unmodified Mimikatz
- Can be bypassed by attackers who modify/obfuscate their tools
- Should be part of defense-in-depth — not treated as a standalone solution
- Definitions are updated constantly; keep Windows updated

> Defender + AppLocker + Group Policy hardening + patch management + user awareness = defense-in-depth. No single tool is enough.
