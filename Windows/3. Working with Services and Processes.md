# Windows Services & Processes 

## 1. Windows Services

### What are Services?

- Services are **long-running background processes** in Windows.
- They can:
    - Start automatically at system boot
    - Run without user interaction
    - Continue running even after logout
- Used for core OS functionality and applications.

### Examples of what services handle:

- Networking
- System diagnostics
- User authentication
- Windows updates
- Application-level tasks (e.g., monitoring tools)
### Service Management

#### GUI Method:

- Managed via **Service Control Manager (SCM)**
- Access using:
    
    services.msc
    
- Displays:
    - Name
    - Description
    - Status
    - Startup Type
    - Run-as user

#### Command Line / PowerShell:

- Using `sc.exe`
- Using PowerShell:
    ```
    Get-Service | ? {$_.Status -eq "Running"} | select -First 2 | fl
    ```
    
### Service States

- Running
- Stopped
- Paused
- Starting / Stopping (transitional states)

### Startup Types

- Automatic
- Manual
- Delayed Start

### Service Categories

- Local Services
- Network Services
- System Services

- Only **administrators** can usually:
    - Create
    - Modify
    - Delete services
- Misconfigurations in services = **common privilege escalation vector**

### Critical Windows Services (Do NOT Kill)

|Service|Function|
|---|---|
|`smss.exe`|Session Manager (handles system sessions)|
|`csrss.exe`|Client-Server Runtime (user-mode Windows subsystem)|
|`wininit.exe`|Starts system initialization processes|
|`logonui.exe`|User login interface|
|`lsass.exe`|Authentication & security policy|
|`services.exe`|Controls service start/stop|
|`winlogon.exe`|Handles login, screen lock|
|`System`|Windows kernel process|
|`svchost.exe (RPCSS)`|Hosts services (RPC-based)|
|`svchost.exe (DCOM/PnP)`|Hosts services (DCOM & Plug and Play)|

Important:

- Some services require **system reboot** if modified.

## 2. Processes

### What are Processes?

- Programs running in memory.
- Can be:
    - OS-related (critical)
    - Application-related (non-critical)

### Critical Processes Examples

- System
- System Idle Process
- Windows Logon Application
- Client Server Runtime (`csrss.exe`)
- Session Manager (`smss.exe`)
- Service Host (`svchost.exe`)
- LSASS (`lsass.exe`)

## 3. LSASS 

### Local Security Authority Subsystem Service (`lsass.exe`)

- Core security component.
### Responsibilities:

- Verifies user logins
- Creates **access tokens**
- Handles password changes
- Logs authentication events (Security Log)

## Service Permissions

Services are long-running background processes, and most sysadmins don't think of them as a security risk — but they absolutely are. Misconfigured service permissions can be used to load malicious DLLs, run programs without admin access, escalate privileges, and maintain persistence on a system. Most of these problems come from bad third-party software installs or simple admin mistakes.

### The Service Account Problem

When a service gets installed, it often gets configured to run under whatever account was logged in during installation. So if Bob installs DHCP while logged in as himself, DHCP now runs as Bob. When Bob leaves the company and his account gets disabled, DHCP stops working — no more IP address leases for anyone on the network.

The fix is simple in concept — always create a dedicated **service account** for critical services. That account exists only to run that service, nothing else. This also follows the **principle of least privilege** — give the account only the permissions it absolutely needs and nothing more.

### Built-in Service Accounts in Windows

Windows has three built-in accounts commonly used for services:

- **LocalService** — limited privileges, acts as an anonymous user on the network
- **NetworkService** — similar to LocalService but can authenticate on the network
- **LocalSystem** — the most powerful, has full access to the local machine

Most services default to **LocalSystem**, which is the highest privilege level possible on a Windows machine. Not every service needs that level of access, so always check what a service actually needs before leaving it on LocalSystem.

### Examining Services with services.msc

Open `services.msc` to view and manage services through the GUI. When you open a service's properties, pay attention to these key fields:

**General tab** — service name (important for command-line use), display name, path to the executable, startup type, and current status.

**Log On tab** — shows what account the service runs under. This is where you'd see LocalSystem or a custom service account configured.

**Recovery tab** — this is a sneaky one. You can configure what happens when a service fails — including running a program. An attacker who can modify this setting could configure a malicious executable to run automatically whenever a legitimate service crashes.

The **path to executable** field is also worth watching. If the NTFS permissions on that directory are weak, an attacker could replace the legitimate executable with a malicious one.

### Examining Services with sc

`sc` is the command-line service controller. It's more useful than `services.msc` for scripting and quick analysis.

**Query a service's configuration:**

```
sc qc wuauserv
```

Output shows you the binary path, start type, what account it runs under, and dependencies. Very useful for spotting if something has been tampered with.

**Query a service on a remote machine:**

```
sc \\hostname or ip query ServiceName
```

**Stop a service:**

```
sc stop wuauserv
```

Without admin privileges this fails with `Access is denied.` You need an elevated command prompt.

**Modify a service's binary path** (this is what an attacker would do):

```
sc config wuauserv binPath=C:\Winbows\Perfectlylegitprogram.exe
```

If you run `sc qc wuauserv` again after this, you'll see the binary path has changed. This is exactly how service binary path hijacking works as an attack.

`sc` is much better than `services.msc` for malware investigations because you can quickly query, compare, and script checks against all services — and it works over the network too.

### Reading Service Permissions with sc sdshow

```
sc sdshow wuauserv
```

Output looks like gibberish at first:

```
D:(A;;CCLCSWRPLORC;;;AU)(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;BA)
(A;;CCDCLCSWRPWPDTLOCRSDRCWDWO;;;SY)S:(AU;FA;CCDCLCSWRPWPDTLOSDRCWDWO;;;WD)
```

This is actually **SDDL — Security Descriptor Definition Language**. Every securable object in Windows has a security descriptor, and this is how it looks in raw form. Here's how to read it:

Each entry in parentheses is one **Access Control Entry (ACE)**. The format is:

```
(A;;CCLCSWRPLORC;;;AU)
 ^  ^^^^^^^^^^^^^^  ^^
 |       |           |
Allow  Permissions  Who (AU = Authenticated Users)
```

Breaking down the permission codes for the `AU` (Authenticated Users) entry:

|Code|Full Name|Meaning|
|---|---|---|
|CC|SERVICE_QUERY_CONFIG|Can query service config|
|LC|SERVICE_QUERY_STATUS|Can query service status|
|SW|SERVICE_ENUMERATE_DEPENDENTS|Can list dependent services|
|RP|SERVICE_START|Can start the service|
|LO|SERVICE_INTERROGATE|Can query current status|
|RC|READ_CONTROL|Can read the security descriptor|

The three entries in the full output represent three different security principals — Authenticated Users (AU), Built-in Administrators (BA), and the System account (SY). Each has different permissions applied.

### Examining Service Permissions with PowerShell

`Get-Acl` gives you a much cleaner and more readable view than `sc sdshow`:

```powershell
Get-ACL -Path HKLM:\System\CurrentControlSet\Services\wuauserv | Format-List
```

Output shows you exactly who has what access in plain English:

```
Owner  : NT AUTHORITY\SYSTEM
Access : BUILTIN\Users Allow ReadKey
         BUILTIN\Administrators Allow FullControl
         NT AUTHORITY\SYSTEM Allow FullControl
```

The advantage over `sc sdshow` is that `Get-Acl` resolves the SIDs into actual account names, making it far easier to read. It also shows the SDDL at the bottom if you need it.

Understanding service permissions matters because services are a commonly overlooked attack surface. The key things to watch for are services running under overly privileged accounts, weak NTFS permissions on the directories containing service executables, and misconfigured recovery actions that could run arbitrary programs. Knowing how to check all of these from the command line means you can script it, automate it, and apply it across an entire network — not just one machine at a time.
