## File Ownership Basics
Every file has **two owners**:

- **User owner** (linked to UID)
    
- **Group owner** (linked to GID)
    

>  **The OS tracks ownership by numbers (UID/GID), not names! If you delete a user, files show the UID number instead of a username.

### View Your Identity
```
id              # Shows UID, GID, and all group memberships
groups          # Quick view of your groups
```

### User Private Groups (UPG)

When a user and their primary group have matching names/numbers:
```
uid=1001(sysadmin) gid=1001(sysadmin)
```

## Changing Your Primary Group

### The `newgrp` Command
```
newgrp group_name    # Opens a new shell with different primary group
exit                 # Back to original shell/group
```

**Example workflow:**
```
$ id
uid=1001(sysadmin) gid=1001(sysadmin) 
$ newgrp research
$ id                 # Now primary group is research
$ touch research-file.txt  # New file owned by research group
$ exit               # Back to normal
```

## Changing File Ownership

### Change Group Only (`chgrp`)
```
chgrp group file           # Change group ownership
chgrp -R group directory/  # Recursive change
```

**Rules:**

- Regular users can only change to groups they belong to
    
- Must own the file (or be root)
### Change User/Group (`chown`) - Root Only
```
chown user file                    # Change only user
chown user:group file              # Change both (colon)
chown user.group file              # Change both (dot) 
chown :group file                  # Change only group
chown .group file                  # Change only group (alternative)
```

## Permission Structure

### The Magic 10 Characters
```
-rw-r--r-- 1 root root 4135 May 27 21:08 /etc/passwd
↑↑↑↑↑↑↑↑↑↑
│└┬┘└┬┘└┬┘
│ │  │  └─ Others permissions
│ │  └──── Group permissions  
│ └─────── User permissions
└───────── File type
```

### File Types (First Character)

|Char|Meaning|Where to find|
|---|---|---|
|`-`|Regular file|Everywhere|
|`d`|Directory|Your folders|
|`l`|Symbolic link|Shortcuts|
|`b`|Block device|`/dev/sda`|
|`c`|Character device|`/dev/tty`|
|`p`|Pipe|Inter-process comms|
|`s`|Socket|Network services|

### Permission Meanings
#### For Files 

- **r** (read): View/copy contents
    
- **w** (write): Modify contents
    
- **x** (execute): Run as program
    
#### For Directories 

- **r** (read): List names only
    
- **w** (write): Add/remove files (needs x!)
    
- **x** (execute): `cd` into it, access contents
    

>  **Write on directory needs execute to work!

##  Useful Commands Reference

### Viewing Ownership
```
ls -l file              # Standard view
ls -la                  # Include hidden files
stat file               # Detailed info (shows UID/GID numbers)
```

### Changing Ownership
```
# As regular user
chgrp research myfile   # If you're in research group

# As root
chown alice myfile
chown alice:devs myfile
chown :devs myfile
chown -R alice:devs /home/alice  # Recursive
```

## Permission Groups (Who gets what)

When accessing a file, Linux checks in this order:

1. **User owner** - Are you the file owner? Use these permissions.
    
2. **Group owner** - Are you in the file's group? Use these permissions.
    
3. **Others** - Everyone else.

## Permission Scenarios: Where Theory Meets Reality

### The Big Picture: Path Matters!

When accessing a file, Linux checks **every directory** in the path, from root (`/`) all the way to the file itself. It's like a series of gates - you need the right key for each one.
```
/          →  /data     →  /data/abc.txt
(access)      (access)      (file perms)
```

## Scenario #1: The "Can't Get There From Here" Trap

**The Setup:**
```
drwxr-xr-x. 17 root root 4096 23:38 /
drwxr-xr--. 10 root root 128  03:38 /data   # Others: r--
-rwxr-xr--.  1 bob  bob  100  21:08 /data/abc.txt  # Bob: rwx
```

**Question:** What access does bob have to `abc.txt`?

**Answer:** **NONE!** 

**Why:** Bob has full `rwx` on the file itself, but...

- To reach `/data/abc.txt`, bob must pass through `/data`
    
- `/data` permissions for others = `r--` (no execute!)
    
- No `x` on directory = can't `cd` into it or use it in paths
    
- Game over before it begins
    

> **Lesson:** Directory execute (`x`) is the **key to the door**. Without it, you can't enter, no matter what file permissions say.

## Scenario #2: The "I Can See but Not Touch" Situation

**The Setup:**
```
drwxr-xr-x. 17 root root 4096 23:38 /
drwxr-xr--. 10 root root 128  03:38 /data   # Others: r--
-rwxr-xr--.  1 bob  bob  100  21:08 /data/abc.txt
```

**Question:** Who can run `ls /data`?

**Answer:** **All users!**

**Why:**

- `ls /data` only needs `r` on the directory
    
- `/` has `x` for everyone (can use in path)
    
- `/data` has `r` for everyone (can list names)
    

**But wait...** try `ls -l /data`:

- Only root and group members (need `x` for detailed view)
    
- Others just see names, no details
    

>  **Lesson:** `r` = list names, `r` + `x` = see details

## Scenario #3: The Deletion Deception

**The Setup:**
```
drwxr-xr-x. 17 root root 4096 23:38 /
drwxrw-rw-. 10 root root 128  03:38 /data   # Everyone: rw- (no x!)
-rwxr-xr--.  1 bob  bob  100  21:08 /data/abc.txt
```

**Question:** Who can delete `/data/abc.txt`?

**Answer:** **Only root!**

**Mind-bending truth:**

- File permissions on `abc.txt`? **Irrelevant** for deletion!
    
- Delete requires `w` on the **directory** (everyone has this!)
    
- But... need `x` on directory to actually get there (only root has this!)
    

>  **Lesson:** Deleting a file is about the **directory's permissions**, not the file's. But you still need `x` to reach it!

## Scenario #4: The "Just Give Me Access" Minimalist

**The Setup:**
```
drwxr-xr-x. 17 root root 4096 23:38 /
dr-xr-x--x. 10 root root 128  03:38 /data   # Others: --x (execute only!)
-rwxr-xr--.  1 bob  bob  100  21:08 /data/abc.txt  # Bob: r-x
```

**Question:** Can bob run `more /data/abc.txt`?

**Answer:** **YES!**

**The path check:**

1. `/` → bob has `x` (can use in path) 
    
2. `/data` → bob has `x` (can enter) 
    
3. `/data/abc.txt` → bob has `r` (can read) 
    

**Even though:**

- Bob can't `ls /data` (no `r` on directory)
    
- Bob can't see what's in there
    
- But if he knows the filename exists, he can access it!
    

>  **Lesson:** `x` alone on a directory = "I can get in, but I'm blind"

## Scenario #5: The Group Puzzle

**The Setup:**
```
drwxr-xr-x. 17 root root    4096 23:38 /
dr-xr-x---. 10 sue  payroll 128  03:38 /data   # bob is NOT in payroll
-rwxr-xr--.  1 bob  bob     100  21:08 /data/abc.txt
```

**Question:** Can bob access `abc.txt`?

**Answer:** **NO!**

**Why it's tricky:**

- Bob owns the file (full `rwx` for him)
    
- But `/data` permissions for "others" = `---` (no access!)
    
- Bob is neither `sue` (owner) nor in `payroll` (group)
    
- Can't even reach the file
    

>  **Lesson:** Directory access trumps file ownership. You can own the file but be locked out of the directory containing it!

## Scenario #6: The Permission Paradox

**The Setup:**
```
drwxr-xr-x. 17 root root 4096 23:38 /
dr-xr-x---. 10 bob  bob  128  03:38 /data   # bob owns it!
----rw-rwx.  1 bob  bob  100  21:08 /data/abc.txt  # File perms: --- for owner?!
```

**Question:** Can bob access his own file?

**Answer:** **NO!** (and yes, it's as weird as it looks)

**The madness explained:**

- Bob owns the file, so **only** user owner permissions apply
    
- User owner permissions = `---` (literally no access!)
    
- Group and others have full access, but bob? Locked out!
    

>  **The Golden Rule:** Owner permissions are checked FIRST. If you're the owner, that's ALL that's checked. Group/others permissions don't matter!

## Changing Permissions: The `chmod` Command

### Two Ways to Rule Them All
```
chmod new_permission file_name
```

 **You must either own the file OR be root to change permissions
## Method 1: Symbolic Method (The Surgical Approach)

Perfect for tweaking specific permissions while leaving others alone.

### The Recipe: **Who + Operator + What**
| Who (Group) | Operator | Meaning          | Permission | What it grants    |
|-------------|----------|------------------|------------|-------------------|
| `u`         | `+`      | add              | `r`        | read              |
| `u`         | `-`      | remove           | `w`        | write             |
| `g`         | `+`      | add              | `w`        | write             |
| `g`         | `-`      | remove           | `r`        | read              |
| `o`         | `=`      | set exactly      | `x`        | execute           |
| `a`         | `+` / `-`| add / remove     | `rwx`      | read/write/execute|
| `a`         | `=`      | set exactly      | —          | all (ugo)         

### Real-World Examples

**Before we start:**
```
-rw-r--r-- 1 root root 0 Dec 19 18:58 abc.txt
```

**Add write for group:**
```
chmod g+w abc.txt
-rw-rw-r-- 1 root root 0 Dec 19 18:58 abc.txt  # Group now has write!
```

**Multiple changes at once:**
```
chmod ug+x,o-r abc.txt
-rwxrwx--- 1 root root 0 Dec 19 18:58 abc.txt  
# User+group: +execute, Others: -read
```

**Set exact permissions with `=`:**
```
chmod u=rx abc.txt
-r-xrwx--- 1 root root 0 Dec 19 18:58 abc.txt  
# User: exactly r-x (write removed!)
```

`=` is powerful - it sets what you specify and REMOVES everything else for that group!

## Method 2: Numeric/Octal Method (The "All or Nothing")

Perfect for setting all permissions at once. Each permission has a number:

|Value|Permission|
|---|---|
|4|Read|
|2|Write|
|1|Execute|

### The Magic Numbers (0-7)

| Number | Permission | Binary |
| ------ | ---------- | ------ |
| 7      | rwx        | 111    |
| 6      | rw-        | 110    |
| 5      | r-x        | 101    |
| 4      | r--        | 100    |
| 3      | -wx        | 011    |
| 2      | -w-        | 010    |
| 1      | --x        | 001    |
| 0      | ---        | 000    |
### The Formula: **UGO** (User-Group-Others)
```
chmod 754 abc.txt  # That's 7 (user) + 5 (group) + 4 (others)
```

Breakdown:

- **7** (user) = 4+2+1 = rwx
    
- **5** (group) = 4+0+1 = r-x
    
- **4** (others) = 4+0+0 = r--
    

Result:
```
-rwxr-xr-- 1 root root 0 Dec 19 18:58 abc.txt
```

## The `stat` Command: Your Permission Detective

Better than `ls -l` for detailed info:
```
stat /tmp/filetest1
```

Output shows BOTH symbolic and octal:
```
Access: (0664/-rw-rw-r--)  Uid: (1001/sysadmin)   Gid: (1001/sysadmin)
           ↑↑↑↑              ↑↑↑
        octal    symbolic
```

Use `stat` when you need to see the numeric permissions for copying!

## Default Permissions: The `umask` Command

Every file/directory is born with default permissions. `umask` decides what gets **subtracted**.

### The Default Maximums

|Type|Maximum|Why|
|---|---|---|
|**Files**|666 (rw-rw-rw-)|Never executable by default (security!)|
|**Directories**|777 (rwxrwxrwx)|Need execute to enter|

### How umask Works: **Maximum - umask = Default**

### Check Your umask
```
sysadmin@localhost:~$ umask
0002  # First 0 = octal, then user, group, others
```

### Different Users, Different umasks
```
# Regular user
sysadmin@localhost:~$ umask
0002  # Less restrictive

# Root user
root@localhost:~# umask
0022  # More restrictive (better security for system files)
```

### umask in Action

**Example 1: umask 027**
```
sysadmin@localhost:~$ umask 027

# Files: 666 - 027 = 640 (rw-r-----)
sysadmin@localhost:~$ touch sample
-rw-r-----. 1 sysadmin sysadmin 0 Oct 28 20:14 sample

# Directories: 777 - 027 = 750 (rwxr-x---)
sysadmin@localhost:~$ mkdir test-dir
drwxr-x---. 1 sysadmin sysadmin 4096 Oct 28 20:25 test-dir
```

**Example 2: Common umask Values**

| umask | Files           | Directories     | Use Case         |
| ----- | --------------- | --------------- | ---------------- |
| 000   | 666 (rw-rw-rw-) | 777 (rwxrwxrwx) | INSECURE         |
| 002   | 664 (rw-rw-r--) | 775 (rwxrwxr-x) | Shared group     |
| 022   | 644 (rw-r--r--) | 755 (rwxr-xr-x) | Standard users   |
| 027   | 640 (rw-r-----) | 750 (rwxr-x---) | Restricted group |
| 077   | 600 (rw-------) | 700 (rwx------) |  Private only    |

### Temporary vs Permanent umask

**Temporary (current session only):**
```
umask 027  # Changes until you log out
```

**Permanent (make it stick):**  
Add to `~/.bashrc`:
```
echo "umask 027" >> ~/.bashrc
```
