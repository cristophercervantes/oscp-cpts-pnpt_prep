# Software Philosophy and Linux Origins

## 1. How Software is Built

- **Source Code:** The original, human-readable instructions written by a programmer.
    
- **Machine Code (Binary/Executable):** The version of the program that the computer actually understands (1s and 0s).
    
- **The Compiler:** A special program that translates **Source Code** into **Machine Code**.
    
- **Interpreted Languages:** Languages like Python, Perl, and Bash that do _not_ need a compiler. Instead, an "interpreting program" reads and runs the code line-by-line.
    

## 2. Software Licensing Models

|**Feature**|**Closed Source (Proprietary)**|**Open Source**|
|---|---|---|
|**Visibility**|Users see only the Binary (Machine Code).|Users can see the Source Code.|
|**Modification**|Illegal to change or reverse engineer.|Users are encouraged to modify and improve it.|
|**Security**|You must trust the vendor.|"Many eyes" inspect the code for bugs and viruses.|
|**Philosophy**|Profit and control-centric.|Collaboration and community-centric.|

## 3. The History of UNIX and Linux

Linux was designed to look and feel like **UNIX**, which was the industry standard before Linux existed.

- **1969:** UNIX is created.
    
- **1973:** UNIX is rewritten in **C**, a language that makes it easy to move the software to different types of hardware.
    
- **1984:** 4.2BSD (a version of UNIX) introduces **TCP/IP**, which becomes the foundation of the modern Internet.
    
- **Early 1990s:** Linux development begins, benefiting from the established "UNIX way" of doing things.
    

## 4. Portability and Standards

Because there are many different versions of Linux and UNIX, the industry needs "rules" to make sure software works everywhere.

- **API (Application Programming Interface):** A standard way for programs to talk to the operating system.
    
- **Porting:** The process of converting a program written for one system to run on another.
    
- **Standards Organizations:**
    
    - **IEEE and POSIX:** These groups create the "specifications" (rules).
        
    - If a programmer follows **POSIX** standards, their code can easily be moved between different Linux and UNIX systems.
        

## 5. Why Open Source Won

Initially, big companies were afraid of open source. However, they realized:

1. Open source programs (like Linux) often **outperformed** expensive proprietary ones.
    
2. The best programmers in the world were contributing to open source in their spare time.
    
3. It is more **robust and stable** because a global community shares the responsibility for fixing bugs.

# Software Licensing Models
## 1. The Three Components of Software Acquisition

When you "buy" software, you are dealing with three distinct factors:

1. **Ownership:** Who owns the intellectual property (IP)? (Usually the creator).
    
2. **Money Transfer:** How payment happens (subscription, one-time fee, or free).
    
3. **Licensing:** The "rules" of what you can and cannot do with the software.
    

## 2. Proprietary vs. Open Source: A Comparison

|**Feature**|**Proprietary (e.g., MS Windows)**|**Open Source (e.g., Linux)**|
|---|---|---|
|**Primary License**|**EULA** (End User License Agreement)|**GPLv2** (General Public License)|
|**Distribution**|**Binary/Executable only** (no source code).|**Source Code** must be available.|
|**Usage Rights**|Usually restricted to one computer.|Can be used on any number of devices.|
|**Modification**|Reverse engineering is forbidden.|Users are encouraged to modify/improve code.|
|**Redistribution**|Not allowed (except for backups).|Allowed and encouraged.|

## 3. The GPLv2 (General Public License v2)

This is the license used by **Linus Torvalds** for the Linux kernel. Its key features include:

- **Availability:** Source code must be provided to anyone who asks for it.
    
- **The "Share-Alike" Requirement:** If you modify the code and distribute it, your changes **must** be under the same GPLv2 license so the community benefits.
    
- **Fee Limits:** You cannot charge for the source code itself, only for the actual cost of distribution (like the cost of a USB drive or shipping).
    

## 4. Understanding FOSS

**FOSS** stands for **Free and Open Source Software**.

- **"Free"** refers to **freedom**, not necessarily price.
    
- In FOSS, the creator gives up the right to forbid people from using, viewing, or redistributing the code.
    
- Even though Linus Torvalds owns Linux, the GPLv2 prevents him from stopping anyone else from using it.
    

## 5. The Licensing Landscape

Because software licensing is often a "political" or philosophical choice, different organizations have created their own standard licenses:

- **Academic Licenses:** Created by universities (e.g., **MIT** and **BSD**).
    
- **Foundation Licenses:** (e.g., **Apache Foundation**).
    
- **Free Software Foundation (FSF):** Creates licenses to push the agenda of software user freedom.

# Licensing Organizations & Business Models
## 1. The Two Pillars of Open Source

While both promote open code, they differ in their philosophy regarding "Freedom."

### **A. Free Software Foundation (FSF)**

- **Focus:** Ethical and moral freedom of the user (**Libre**).
    
- **Core Concept: Copyleft:** A method of ensuring that modified versions of free software remain free. If you redistribute changes, you **must** use the same license.
    
- **Key Licenses:** **GPL** (General Public License) and **LGPL** (Lesser GPL).
    
- **GPLv3 vs. Tivoization:** GPLv3 was created to stop "Tivoization"—where a company shares the source code but prevents modified code from running on their specific hardware.
    

### **B. Open Source Initiative (OSI)**

- **Focus:** Practical and business advantages of open code.
    
- **Core Concept: Permissive Licensing:** Licenses that allow you to use open-source code inside a **closed-source/proprietary** product, as long as you give credit.
    
- **Key Licenses:** **BSD** and **MIT**.
    
- **Approach:** They don't write licenses; they "approve" ones that meet the **Open Source Definition**.
    
## 2. Creative Commons (CC)

Designed for **non-software works** (images, text, music, 3D plans). It uses four main "building blocks" to create six standard licenses.

- **BY (Attribution):** Must credit the creator.
    
- **SA (ShareAlike):** Must use the same license for modifications (Copyleft).
    
- **NC (Non-Commercial):** Cannot use for-profit.
    
- **ND (No-Derivatives):** Can share, but cannot change the work.
    
- **CC0:** No rights reserved (Public Domain).
    

|**License**|**Description**|
|---|---|
|**CC BY**|Most permissive; just give credit.|
|**CC BY-SA**|Give credit and share changes under the same terms.|
|**CC BY-NC**|Credit required; no commercial use allowed.|
|**CC BY-ND**|Credit required; work must remain original/unchanged.|

## 3. Open Source Business Models

"Free" refers to **freedom**, not **price**. You can legally sell GPL software, but most money is made through "Value-Add" services.

- **Support & Warranty:** Selling expertise and technical help (e.g., **Red Hat**, **Canonical/Ubuntu**).
    
- **Hardware Integration:** Selling a physical device that runs on Linux (e.g., **TiVo**, IoT cameras, Smart TVs).
    
- **Professional Services:** Creating a free tool (e.g., **Wireshark**) and then selling custom features or consulting.
    
- **Cloud Services:** Providing the software as a hosted service (SaaS), where the user pays for the convenience of the cloud rather than the code.
