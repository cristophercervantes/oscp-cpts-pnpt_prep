# Intro
## **Why Create Groups?**

In a nutshell: **Groups = Collaboration.** If five people need to work on the "Research" project, you don't give them individual permissions one by one. You put them in a group called `research`, give that group access to the folder, and you’re done.

### **Checking Your Groups**

Once you’ve made a change, you need to verify it. You have two main tools:

- **`grep`**: Best for a quick look at the local `/etc/group` file.
    
- **`getent`**: Best for professional environments where users might be stored on a network server (not just your local machine).
    

## **Creating a Group (`groupadd`)**

Only the **root** user (or someone using `sudo`) can create groups.

- **Simple creation:** `groupadd development` (The system picks the next available ID number).
    
- **Pick a specific ID:** `groupadd -g 1005 research` (Useful if you want to keep your ID numbers organized).
    
- **System groups:** `groupadd -r sales` (The `-r` flag creates a "system" group with a low ID number, usually below 1000).
    

## **The "Matching ID" Problem**

In many Linux versions (like Red Hat), the system tries to keep a user's **UID** and their private group's **GID** identical (e.g., User 1001 is in Group 1001).

> **Pro-Tip:** Don't manually pick GIDs in the same range as your UIDs. If you take GID 1005 for a "Research" group, you might prevent a future user from having a matching UID/GID pair.

## **Group Naming: The Unwritten Rules**

Linux is flexible with names, but being "too creative" can break things. Follow these rules to keep your system stable:

1. **Start right:** Use a lowercase letter (`a-z`) or an underscore (`_`).
    
2. **Length:** Keep it under **16 characters**. While 32 is technically allowed, it causes headaches on older systems.
    
3. **The Middle:** Use letters, numbers, dashes (`-`), or underscores (`_`).
    
4. **The End:** Never end a group name with a hyphen (`-`).

## **Modifying a Group (`groupmod`)**

Think of a group like a folder with a ID number on it. You can change the label (name) easily, but if you change the ID number, the system loses track of what belongs inside.

### **Changing the Name (`-n`)**

- **Command:** `groupmod -n new_name old_name`
    
- **Result:** Very safe. Since Linux looks at the **GID** (the number) to determine permissions, changing the name doesn't break anything.
    
- **Example:** If you rename `sales` to `clerks`, every file previously owned by `sales` will automatically show `clerks` as the owner.
    

### **Changing the GID (`-g`)**

- **Command:** `groupmod -g 2000 group_name`
    
- **Result:** **Dangerous.** If you change the GID number, all existing files owned by that group become **Orphaned**.
    
- **What happens:** The files still have the old GID number (e.g., 491), but because no group name is linked to 491 anymore, the system just shows the number instead of a name.
    

> **Pro-Tip:** If you have orphaned files, find them using:
> 
> `find / -nogroup`

## **16.2.3 Deleting a Group (`groupdel`)**

Before you hit delete, you need to check two things: **Orphans** and **Primary Status**.

### **The "Primary Group" Rule**

You **cannot** delete a group if it is currently set as any user's **Primary Group**.

- **The Fix:** You must first modify the user to give them a different primary group. Once the group is "empty" of primary members, you can delete it.
    

### **The "Orphan" Problem**

Just like changing a GID, deleting a group leaves all its files behind as orphans.

- Before deleting, you should ideally transfer those files to a new group or be prepared to clean them up.

## **Creating Users: The "Safety First" Approach**

While you _could_ technically create a user by typing a new line into `/etc/passwd` and `/etc/shadow`, it’s a bad idea. One typo could lock everyone out of the system.

Instead, we use the `useradd` command. It acts like an automated assistant that follows a specific set of rules (defaults) to build the account for you safely.

## **16.3.1 Viewing and Changing Defaults (`useradd -D`)**

To see what rules the system currently follows when creating a user, type `useradd -D`. These settings live in `/etc/default/useradd`.

### **Key Default Values Explained**

- **GROUP:** The "fallback" group if you don't specify one (usually GID 100).
    
- **HOME:** The "neighborhood" where new user folders are built (usually `/home`).
    
- **INACTIVE:** How many days to give a user to fix an expired password before locking the account entirely. `-1` means this safety net is turned off.
    
- **SHELL:** Which "language" the terminal speaks when they log in (usually `/bin/bash`).
    
- **SKEL (`/etc/skel`):** The "Starter Kit." Everything in this folder is automatically copied into a new user's home directory (like `.bashrc` or desktop settings).
    

> **Pro-Tip:** To change a default without opening a text editor, use:
> 
> `useradd -D -f 30` (This sets the default "Inactive" grace period to 30 days).

## **16.3.2 The Master Policy File (`/etc/login.defs`)**

While `useradd -D` handles the basics of the account, `/etc/login.defs` handles the **security policy**. This file defines the "big picture" rules for UIDs and Passwords.

### **Security & Identity Policies**

|**Setting**|**What it controls**|**Why it matters**|
|---|---|---|
|**PASS_MAX_DAYS**|Max password age|Forces users to rotate passwords for better security.|
|**PASS_MIN_DAYS**|Min wait to change|Prevents users from immediately switching back to an old password.|
|**UID_MIN / GID_MIN**|Starting ID number|Keeps "Human" accounts (500+) separate from "System" accounts (1-499).|
|**CREATE_HOME**|Auto-folder creation|If `yes`, the system builds the `/home/user` folder automatically.|
|**UMASK**|Folder privacy|`077` is like a "Keep Out" sign; it means only the owner can see their files.|
|**USERGROUPS_ENAB**|UPG Support|If `yes`, every "Alice" gets an "Alice" group automatically.|

## **The "Skeleton" Directory: `/etc/skel`**

Think of `/etc/skel` as a **template**.

- If you want every new employee to have a "Company_Policy.txt" file on their desktop the moment they join, you put that file in `/etc/skel`.
    
- When you run `useradd`, Linux copies everything from `/etc/skel` into the new user's home folder and changes the ownership so the user can use it.

## **The "Anatomy" of a New Account**

When you create a user with `useradd`, you aren't just giving them a name; you're building a profile. Here’s what you need to consider:

- **Username:** Follow the same rules as groups (lowercase `a-z`, start with a letter/underscore, max 16–32 chars).
    
- **UID (The ID Number):**
    
    - **0:** Reserved for **root** (The "Master Key").
        
    - **1–999:** Reserved for **System Daemons** (background services).
        
    - **1000+:** Standard range for **Human Users**.
        
- **Home Directory:** The system usually creates `/home/username`.
    
    - Use `-m` to ensure the folder is actually created.
        
    - Use `-d` to specify a custom path (e.g., `/projects/jane`).
        
- **The Shell:** Usually `/bin/bash` for humans. For service accounts, use `/sbin/nologin` (this prevents anyone from actually logging into that account).
    
- **The Comment (`-c`):** This is where you put the human’s full name (e.g., "Jane Doe").
    

## **Putting it Together: The `useradd` Command**

Think of `useradd` as a multi-tool. You can set everything in one single line.

**Example Command:** `root@localhost:~# useradd -u 1009 -g users -G sales,research -m -c 'Jane Doe' jane`

**What just happened?**

1. **Identity:** Jane gets UID **1009**.
    
2. **Groups:** Her main house is **users**, but she has keys to the **sales** and **research** offices.
    
3. **Real Estate:** The system built a folder at `/home/jane` and filled it with "starter files" from `/etc/skel`.
    
4. **Database:** Linux automatically updated `/etc/passwd`, `/etc/shadow`, and `/etc/group`.
    

> ** Note:** After running this, Jane **cannot log in yet**. The account exists, but it has no password!

## **The Art of the Password**

A secure system is only as strong as its weakest password.

### **The "Don'ts" of Passwords**

- **Personal Info:** Never use names, birthdays, pets, or phone numbers.
    
- **Dictionary Words:** Common words are easy for "brute-force" programs to guess.
    

### **The "Dos" of Passwords**

- **Mix it up:** Use a "salad" of Uppercase, Lowercase, Numbers, and Symbols.
    
- **Length vs. Memory:** It should be long enough to be secure, but short enough that the user doesn't write it on a Post-it note stuck to their monitor.
    

### **Password Lifetime (Aging)**

- **The Benefit:** If an attacker steals a password, they only have access until the password expires.
    
- **The Policy:** * **Sensitive/High Risk:** Change every **30 days**.
    
    - **Standard/Low Risk:** Change every **90 days**.
        
    - **Tip:** Don't force changes _too_ often, or users will just choose "Password123", "Password124", etc.

## **Setting a User Password (`passwd`)**

In Linux, the `passwd` command is a "two-way street"—both the admin and the user can use it, but with different levels of power.

### **The Administrator's Power**

- **Command:** `sudo passwd username`
    
- **Rule-Breaking:** The root user (admin) can set any password they want. Even if the system says "BAD PASSWORD" (too short or too simple), the admin can ignore the warning and force it anyway.
    
- **Initial Setup:** When you create a new account like `jane`, she can't log in until you run `passwd jane` to give her a starting password.
    

### **The User's Power**

- **Command:** `passwd` (no username needed).
    
- **Self-Service:** Users can change their own password at any time, but they **must** know their current password first.
    
- **Strict Rules:** Unlike the admin, users cannot ignore safety warnings. If the password is too simple, the system will reject it. They usually get **three tries** before the command kicks them out.
    

## **Managing Password Aging (`chage`)**

The `chage` command (short for **ch**ange **age**) is the "Calendar Manager" for passwords. It allows an admin to decide exactly when a password expires or when a user needs a nudge to change it.

### **The `chage` Cheat Sheet**

|**Goal**|**Option**|**Practical Example**|
|---|---|---|
|**Check Status**|`-l`|`chage -l jane` (Shows a readable summary of all dates).|
|**Set Expiry**|`-M`|`chage -M 60 jane` (Forces a change every 60 days).|
|**Set Warning**|`-W`|`chage -W 7 jane` (Start nagging them 7 days before expiry).|
|**Grace Period**|`-I`|`chage -I 30 jane` (The account locks 30 days _after_ expiry if ignored).|
|**Account Death Date**|`-E`|`chage -E 2026-12-31 jane` (Account stops working on this date).|

## **Modifying and Deleting Users**

People change roles, and people leave companies. Linux gives you two tools to handle this:

### **1. Modifying Users (`usermod`)**

This is the "edit" button for an account.

- **Locking an Account:** `usermod -L jane` (Locks her out without deleting her files—perfect for temporary leave).
    
- **Unlocking:** `usermod -U jane`.
    
- **Changing Groups:** `usermod -G marketing jane` (Adds her to the marketing group).
    

### **2. Deleting Users (`userdel`)**

- **Standard Delete:** `userdel jane` (Removes the account from the database, but **leaves her home directory** behind).
    
- **The "Clean Sweep":** `userdel -r jane` (Deletes the account **and** wipes her home directory and mail). Use this with caution!
