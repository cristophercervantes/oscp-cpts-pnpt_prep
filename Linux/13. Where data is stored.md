# Introduction to GNU/Linux

### 1. The GNU/Linux Distinction

Most people say "Linux," but technically it is **GNU/Linux**:

- **GNU:** The "wrapper." It provides the free software, utilities, and open-source versions of UNIX commands.
    
- **Linux Kernel:** The "engine." It is the core of the OS that manages hardware and software interactions.
    

### 2. The Role of the Kernel

The kernel is loaded at **boot time** and remains active until shutdown. Its primary job is to act as the intermediary between the user/apps and the hardware.

**Key Subsystems/Functions:**

- **System Call Interface:** How programs talk to the kernel.
    
- **Process Management:** Handling running programs.
    
- **Memory Management:** Allocating RAM efficiently.
    
- **Virtual Filesystems:** Managing how data is stored/retrieved.
    
- **Networking & Device Drivers:** Controlling hardware like Wi-Fi cards and GPUs.
    

### 3. How the System Works

1. **The Shell:** The user enters a command into a shell (terminal).
    
2. **The Kernel:** Receives the command and manages the necessary processes.
    
3. **Resource Access:** The kernel grants those processes access to hardware (RAM, Disk, Network, Monitor).
    

### 4. File System Standards

Because Linux has thousands of files, it follows a specific map:

- **FHS (Filesystem Hierarchy Standard):** The official guideline for where files should live (e.g., `/bin`, `/etc`).
    
- **Pseudo Filesystems:** Special directories that don't exist on a disk but represent system info:
    
    - `/proc`: Information about running processes and the kernel.
        
    - `/sys`: Information about hardware and device drivers.

# Process & Pseudo Filesystems
Linux uses **pseudo filesystems** that exist only in memory (RAM), not on the physical disk, to provide real-time system data.

- **/proc**: Information about running processes (organized by PID), hardware, and kernel config.
    
- **/sys**: Information about hardware devices and drivers.
    
- **/dev**: Access point for hardware devices (represented as files).
    

### Key Files in `/proc`

|**File**|**Contents**|
|---|---|
|`/proc/cmdline`|Parameters passed to the kernel at boot.|
|`/proc/meminfo`|Detailed RAM usage statistics.|
|`/proc/modules`|List of currently loaded kernel modules (drivers).|
|`/proc/sys/`|**Writable** files; changing these modifies kernel behavior instantly (temporary until reboot).|

##  Process Hierarchy

Every process has a "family" relationship:

- **PID 1**: The first process started by the kernel (`init` or `systemd`). All other processes are descendants of PID 1.
    
- **Parent Process (PPID)**: The process that started another.
    
- **Child Process**: The process that was started.
    
- **PID Max**: When the system reaches its maximum PID, it "rolls over" and starts over from the bottom.
    

**Commands to view hierarchy:**

- `pstree`: Displays a visual tree of parents and children.
    
- `ps --forest`: Adds ASCII art connectors to show relationships in a list.
    

## Viewing Snapshots (`ps`)

The `ps` command gives a **static** view of what is happening at that exact microsecond.

- `ps`: Shows processes in the _current_ terminal.
    
- `ps aux`: Shows every process on the system (BSD style).
    
- `ps -ef`: Shows every process with Parent PID info (UNIX style).
    
- **Pro Tip**: Use `ps aux | grep [name]` to find a specific program.
    

## 13.2.3 Real-Time Monitoring (`top`)

The `top` command provides a dynamic, updating list of processes, usually sorted by **% CPU usage**.

### Interactive Keys in `top`:

- **`k`**: Kill a process (requires PID and signal; `9` is a forced kill).
    
- **`r`**: "Renice" (change priority). Values range from **-20** (highest priority) to **19** (lowest).
    
- **`q`**: Quit `top`.
    
- **`h`**: Help menu.
    
### Understanding "Load Average"

Found at the top of `top` or via the `uptime` command. It shows three numbers (1, 5, and 15-minute intervals):

- **1.0 on a 1-core CPU** = 100% capacity.
    
- **1.0 on a 4-core CPU** = 25% capacity.

# Memories
## Memory Concepts

### 1. Virtual Addressing

Linux uses **Virtual Addressing** to manage physical RAM.

- **The Illusion:** Every process thinks it has access to a massive, contiguous block of memory, even if that memory is actually fragmented across physical RAM or stored on a disk.
    
- **The Benefit:** It prevents processes from conflicting. Process A cannot "see" or "corrupt" the memory of Process B because the kernel maps their virtual addresses to different physical locations.
    
- **Blocks:** Memory is managed in equally sized units called **blocks** (or pages).
    

### 2. User Space vs. Kernel Space

To maintain stability, Linux divides memory into two distinct zones:

- **Kernel Space:** * Reserved for the Kernel's core code and drivers.
    
    - Highly protected; regular programs cannot touch this area.
        
    - This isolation prevents a buggy app (like a web browser) from crashing the entire computer.
        
- **User Space:** * Where your applications (Office, Shell, Games) live.
    
    - Apps must use **System Calls (APIs)** to ask the Kernel for hardware resources.
        

##  Monitoring Memory with `free`

The `free` command is the standard tool for a quick memory health check.

### Viewing the Output

By default, `free` shows values in **bytes**, which is hard to read. Use these flags:

- `free -m`: Display in Megabytes (MB).
    
- `free -g`: Display in Gigabytes (GB).
    
- `free -s 10`: Update the stats every 10 seconds (useful for monitoring a spike).
    

### Understanding the `free` Table

|**Column/Row**|**Description**|
|---|---|
|**total**|Total physical RAM installed on the system.|
|**used**|RAM currently being used by the system and apps.|
|**free**|RAM that is completely empty and untouched.|
|**buff/cache**|RAM used by the kernel to speed up disk access. **Note:** This memory is technically "available" because the kernel will give it up if an app needs it.|
|**Swap**|"Virtual Memory" located on the hard drive.|

##  3. Swap Space (Virtual Memory)

**Swap** is a portion of your hard drive or SSD that acts as "overflow RAM."

- **The Good:** It prevents the system from crashing immediately when RAM is 100% full.
    
- **The Bad:** Hard drives (even SSDs) are much slower than RAM. If the system starts using Swap heavily (called "swapping"), performance will drop significantly.
    
- **The Danger:** If both RAM and Swap are exhausted, the **OOM Killer (Out Of Memory Killer)** will automatically start terminating processes to prevent a total system freeze.

# Log Files
### 1. The Logging Daemons

A "daemon" is a background process. Over time, the tools that manage logs have evolved:

- **The Legacy Way:** Two separate daemons, `syslogd` (system) and `klogd` (kernel).
    
- **The Common Way (`rsyslogd`):** A modern, single daemon that handles both system and kernel logs.
    
- **The Modern Way (`journald`):** Part of **systemd**. It captures logs in a **binary format**, which allows for faster searching and more metadata.
    

###  2. Where Logs Live: `/var/log`

Almost all logs are stored in the `/var/log` directory. Here are the most critical files you need to know:

|**File Name**|**Purpose / Content**|
|---|---|
|**`dmesg`**|Kernel messages from **boot up** (hardware detection, drivers).|
|**`secure`** (or `auth.log`)|Authentication logs (logins, `sudo` usage, failed passwords).|
|**`messages`** (or `syslog`)|General system messages that don't fit elsewhere.|
|**`boot.log`**|Status messages from services starting during boot.|
|**`cron`**|Records of scheduled tasks (cron jobs).|
|**`maillog`**|Activity from the email server/daemon.|
|**`Xorg.0.log`**|Logs from the Graphical User Interface (GUI) server.|

### 3. Log Rotation

Log files grow constantly. To prevent them from filling up the entire hard drive, Linux uses **Log Rotation**:

1. **Renaming:** The current log (e.g., `secure`) is renamed with a date or number (e.g., `secure-20260217`).
    
2. **Replacement:** A fresh, empty `secure` file is created for new data.
    
3. **Compression:** Older rotated logs are often compressed (ending in `.gz`) to save space.
    

### 4. How to View Logs

You can't always use a simple text editor for logs because some are **binary data**, not plain text.

### Text-Based Logs

Use standard commands for these:

- `cat` / `less`: Good for reading through a file.
    
- `tail -f /var/log/messages`: **Crucial Command.** "Follows" the file in real-time, showing new lines as they are written.
    

### Binary Logs

If the `file` command identifies a log as **"data"**, it is likely binary. Using `cat` on these will result in gibberish on your screen.

- **`journalctl`**: The primary tool to view logs managed by `systemd-journald`.
    
- **`last`**: Reads `/var/log/wtmp` to show a history of successful user logins.
    
- **`lastb`**: Reads `/var/log/btmp` to show **failed** login attempts (requires root/sudo).

# Kernel Messages

The Kernel generates messages constantly, especially during boot-up or when hardware is plugged in. Because the Kernel doesn't have a traditional "text file" while it's first loading (since the hard drive might not even be mounted yet), it uses a **Ring Buffer**.

### 1. The Kernel Ring Buffer

- **Definition:** A dedicated area of memory where the Kernel stores its logs.
    
- **"Ring" Nature:** It has a fixed size (set at compile time, often 512KB). Once it is full, new messages **overwrite** the oldest ones.
    
- **Volatility:** Since it's in memory, the buffer is cleared if the system loses power or reboots.
    

### 2. The `dmesg` Command

The `dmesg` command is the primary tool used to "dump" the contents of the Kernel ring buffer to your screen.

- **Usage:** Often produces a massive wall of text. It is standard practice to pipe it to `less` or `grep`.
    
- **Common Use Case:** Troubleshooting hardware (USB drives, Network cards, Disk errors).
    
- **Case Insensitivity:** Use the `-i` flag with `grep` to find hardware regardless of capitalization.
    
    - _Example:_ `dmesg | grep -i "usb"` or `dmesg | grep -i "eth0"`.
        

###  3. Kernel Logs on Disk

While `dmesg` shows what's in memory _right now_, the system does eventually save these messages to files in `/var/log`:

|**File Path**|**Description**|
|---|---|
|**`/var/log/dmesg`**|A snapshot of kernel messages specifically from the **last boot**.|
|**`/var/log/messages`**|Contains kernel messages _as they happen_ while the system is running, mixed with other app logs.|
|**`/etc/rsyslog.conf`**|The configuration file where you can tell the system to send Kernel messages to a specific, dedicated file if you choose.|

# Filesystem Hierarchy Standard
The FHS is a set of guidelines for where files and directories should be placed. It categorizes directories based on two factors:

1. **Shareable vs. Non-shareable:** Can the files be used by other machines over a network?
    
2. **Static vs. Variable:** Do the files remain unchanged (Static) or do they grow/change (Variable)?
    

### The 4 Main Hierarchies

|**Hierarchy**|**Directory**|**Description**|
|---|---|---|
|**First**|`/` (Root)|Essential for booting and system repair.|
|**Second**|`/usr`|Multi-user utilities and applications (Static/Shareable).|
|**Third**|`/usr/local`|Software compiled manually or not from the official distro.|
|**Fourth**|`/var`|Variable data like logs, mail, and print spools.|

## Essential Root Directories

|**Directory**|**Content / Purpose**|
|---|---|
|**`/bin`**|Essential user binaries (e.g., `ls`, `cp`, `rm`).|
|**`/sbin`**|Essential system binaries for the **root** user (e.g., `fdisk`, `reboot`).|
|**`/etc`**|**Host-specific configuration files.** This is where you change system settings.|
|**`/boot`**|Files needed to start the computer (Kernel, Bootloader).|
|**`/lib`**|Shared libraries (`.so` files) that support binaries in `/bin` and `/sbin`.|
|**`/home`**|Personal storage for users (e.g., `/home/bob`).|
|**`/root`**|The home directory for the **root user** (not inside `/home`).|
|**`/tmp`**|Temporary files (often cleared at boot).|
|**`/media` & `/mnt`**|Mount points for removable media (USB/CD) or manual mounts.|
|**`/opt`**|Optional/Third-party proprietary software.|

##  The Linux Filesystem Map

### FHS Classification Matrix

The FHS isn't just a list of folders; it’s a logic system based on how data behaves.

| **Shareable** (Can be on a network drive) | **Unshareable** (Must be local)    |
|--------------------------------------------|-------------------------------------|
| **Static** (Doesn't change)                | `/usr`, `/opt`                      | `/etc`, `/boot`                     |
| **Variable** (Changes/Grows)               | `/var/mail`, `/var/spool/news`      | `/var/run`, `/var/lock`             |

## Functional Groupings

Instead of memorizing the whole list at once, group them by their **purpose**:

### The "System Vital" Group

These are required to even get the computer to a command prompt.

- **`/` (Root):** The starting point. Everything else is "mounted" under here.
    
- **`/boot`:** Contains the **Linux Kernel** and Grub bootloader files.
    
- **`/etc`:** Think of this as the "Control Panel." It holds text-based config files for the whole system.
    
- **`/lib` & `/lib64`:** The "engine parts" (.so files) that programs need to run.
    

### The "Binary" (Executable) Group

- **`/bin`:** Essential commands for **all users** (`ls`, `nano`, `cat`).
    
- **`/sbin`:** System commands for **root/admin** (`reboot`, `fdisk`, `iptables`).
    
- **`/usr/bin`:** Most of your daily apps (Python, Git, compilers).
    
- **`/usr/local/bin`:** Software you compiled yourself from source.
    

### The "User & Data" Group

- **`/home`:** Where users store their music, docs, and desktop files.
    
- **`/root`:** The "Home" for the administrator. It is kept separate from `/home` for security and maintenance.
    
- **`/tmp`:** The "scratchpad." Files here are usually deleted when you reboot.
    

### The "Hardware & Info" Group

These are **Pseudo Filesystems** (they don't live on your hard drive).

- **`/dev`:** Hardware represented as files (e.g., `/dev/sda` is your hard drive).
    
- **`/proc`:** Real-time info about **Processes**.
    
- **`/sys`:** Real-time info about **Hardware Devices**.
    

## The `/var` Danger Zone

The `/var` directory is unique because its size is unpredictable.

- **Log Files (`/var/log`):** If a service starts erroring rapidly, these files can grow to many gigabytes in minutes.
    
- **Spool Files (`/var/spool`):** Holds print jobs or outgoing emails.
    
- **Risk:** If `/var` fills up your entire disk, the Kernel can't write new data, often leading to a **system crash**. This is why pros often put `/var` on its own physical drive partition.
    

### Comparison with Windows

If you are coming from a Windows background, use this mental map:

- **`/etc`** $\approx$ The Windows Registry.
    
- **`/home`** $\approx$ `C:\Users`.
    
- **`/bin` & `/usr/bin`** $\approx$ `C:\Windows\System32`.
    
- **`/opt`** $\approx$ `C:\Program Files`.
    
- **`.so` files** $\approx$ `.dll` files.
