## Special Permissions: When Basic Just Isn't Enough

Sometimes `rwx` doesn't cut it. Enter the special permissions: **setuid**, **setgid**, and the **sticky bit**. These are the power tools of Linux permissions.

## Setuid (Set User ID) - The "Run As Owner" Permission

### What It Does

When set on an **executable file**, the program runs with the **file owner's privileges**, not the user who ran it.

### The Classic Example: `passwd`
```
# Normal users can't touch /etc/shadow
$ more /etc/shadow
Permission denied

$ ls -l /etc/shadow
-rw-r-----. 1 root root 5195 /etc/shadow  # Only root can read!

# BUT... passwd command works!
$ passwd
Changing password for user...
```

**The Magic:**
```
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 31768 /usr/bin/passwd
#  ↑
#  s in user execute position = setuid!
```

When you run `passwd`:

- You're still `sysadmin`
    
- But `passwd` runs as **root** (the file owner)
    
- So it can modify `/etc/shadow` on your behalf
    

### Spotting Setuid

|Symbol|Meaning|
|---|---|
|`s`|setuid + execute (working correctly)|
|`S`|setuid only, NO execute (broken/useless)|

### Managing Setuid

**Symbolic method:**
```
chmod u+s file     # Add setuid
chmod u-s file     # Remove setuid
```

**Numeric method:**
```
chmod 4755 file    # Add setuid (4xxx) to 755
chmod 0755 file    # Remove setuid (0xxx)
```
**Important:** Setuid on directories? **Ignored by Linux.** Only works on files.

## Setgid (Set Group ID) - The Two-Faced Permission

Setgid does **double duty** - behaves differently on files vs directories.

### Setgid on Files (Like setuid, but for groups)

**What it does:** Program runs with the **file group's** privileges.

**Classic example: `wall` command**
```
$ ls -l /usr/bin/wall
-rwxr-sr-x 1 root tty 30800 /usr/bin/wall
#      ↑
#      s in group execute = setgid!

$ ls -l /dev/tty1
crw--w----. 1 root tty 4, 1 /dev/tty1  # tty group can write
```

When you run `wall`:

- You're not in `tty` group
    
- But `wall` runs with `tty` group privileges
    
- Can write to terminal devices normal users can't touch
    

###  Setgid on Directories (The Collaboration Superpower)

**What it does:** New files inherit the **directory's group**, not the creator's primary group.

#### The Collaboration Problem

**Without setgid:**
```
# Team directory setup
drwxrwx---. 2 root team 4096 /home/team  # Group = team

# Bob creates a file
$ touch /home/team/report.txt
-rw-r-----. 1 bob payroll 0 report.txt  
# ↑ Group = bob's primary group (payroll)
# Sue (in team) can't access it! 
```

**With setgid:**
```
# Add setgid to directory
$ chmod g+s /home/team
$ ls -ld /home/team
drwxrws---. 2 root team 4096 /home/team  # Note the 's'

# Bob creates a file NOW
$ touch /home/team/report.txt
-rw-r-----. 1 bob team 0 report.txt  
# ↑ Group INHERITS from directory!
# Sue can access it! 
```

Subdirectories automatically inherit setgid!
```
$ mkdir /home/team/subproject
$ ls -ld /home/team/subproject
drwxr-s---. 2 bob team 4096 /home/team/subproject  # Auto-setgid!
```

### Spotting Setgid

|Location|Symbol|Meaning|
|---|---|---|
|On files|`s` in group execute|setgid works|
|On dirs|`s` in group execute|inheritance enabled|
|Anywhere|`S`|setgid set but NO execute (broken)|

### Managing Setgid

**Symbolic:**
```
chmod g+s file_or_dir    # Add setgid
chmod g-s file_or_dir    # Remove setgid
```

**Numeric:**
```
chmod 2755 file_or_dir   # Add setgid (2xxx) to 755
chmod 0755 file_or_dir   # Remove setgid (0xxx)
```

## Sticky Bit - The "Don't Touch My Stuff" Permission

### The Problem It Solves

Remember: **Write permission on a directory = delete ANY file in it**, even files you don't own! 
```
# Without sticky bit - chaos!
drwxrwxrwx. 2 root root 4096 /shared
# Alice creates a file
-rw-r--r--. 1 alice alice 100 /shared/notes.txt
# Bob can delete it! (Bob has write on directory)
rm /shared/notes.txt  # Deletes Alice's file! 
```

### The Solution: Sticky Bit

With sticky bit, users can only delete files they own (plus root).

### Classic Examples: `/tmp` and `/var/tmp`
```
$ ls -ld /tmp
drwxrwxrwt. 1 root root 4096 /tmp   
#           ↑ 't' = sticky bit active!

$ ls -ld /var/tmp  
drwxrwxrwt. 1 root root 4096 /var/tmp
```

### Spotting the Sticky Bit

|Symbol|Meaning|
|---|---|
|`t`|sticky bit + others execute (working)|
|`T`|sticky bit only, NO others execute|

>  **Note:** Capital `T` isn't necessarily broken - it just means others don't have execute. This might be intentional!

### Managing the Sticky Bit

**Symbolic:**
```
chmod o+t directory     # Add sticky bit
chmod o-t directory     # Remove sticky bit
```

**Numeric:**
```
chmod 1777 directory    # Add sticky (1xxx) to 777
chmod 0777 directory    # Remove sticky (0xxx)
```

## Links: Multiple Paths to the Same Data

### The Problem
```
# File buried deep in filesystem
/usr/share/doc/superbigsoftwarepackage/data/2013/october/tenth/valuable-info.txt
# You need easy access, but can't move it (other programs depend on it)
```

### Solution: Links!

Two types: **Hard Links** and **Symbolic (Soft) Links**

##  How Files Really Work: The Inode Concept

### The Inode Table (File Metadata)

Every file has:

- **Inode number** (unique identifier on the partition)
    
- **Metadata** (permissions, ownership, timestamps)
    
- **Pointers to data blocks** (where content lives)
    
### Directory Entries: The Name-to-Inode Map
```
$ ls -i /etc
# Inode numbers shown:
123 passwd
175 shadow
144 group
897 gshadow
```

The directory just maps **names → inode numbers**. The actual file data lives elsewhere!

## Hard Links: Multiple Names, Same File

### What They Are

Multiple filenames pointing to the **same inode number**.
```
# Directory entries after creating hard link:
File Name    Inode Number
passwd       123
mypasswd     123   # Same inode!
shadow       175
```

### Creating Hard Links
```
$ echo "important data" > original.txt
$ ls -li original.txt
278772 -rw-rw-r--. 1 sysadmin sysadmin 15 original.txt
#                    ↑ link count = 1

$ ln original.txt hardlink.txt  # Create hard link
$ ls -li original.txt hardlink.txt
278772 -rw-rw-r--. 2 sysadmin sysadmin 15 hardlink.txt
278772 -rw-rw-r--. 2 sysadmin sysadmin 15 original.txt
#                    ↑ link count now = 2!
```

### The Link Count

- Shows how many names point to this inode
    
- Increments with each new hard link
    
- Decrements when files are deleted
    
- File data only disappears when link count reaches 0

## Symbolic (Soft) Links: Shortcuts

### What They Are

A special file that contains a **path to another file**.
```
$ ln -s /etc/passwd mypasswd
$ ls -l mypasswd
lrwxrwxrwx. 1 sysadmin sysadmin 11 mypasswd -> /etc/passwd
# ↑ 'l' = link type
```

### How They Work

- Small file (just stores the path string)
    
- Points to target by **pathname**, not inode
    
- If target moves/deleted, link breaks
