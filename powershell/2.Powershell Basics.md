# PowerShell Basics

## 1.1 What is PowerShell?

PowerShell is **two things at once**:

1. **A management tool** — used to manage Windows, Linux, other Microsoft server products, and cloud platforms like Azure and AWS.
2. **A scripting / programming language** — used to write automation scripts and even build custom GUI applications.

### Historical Context: Why PowerShell Exists

PowerShell has its roots in the history of the PC operating system:

|Era|Technology|Notes|
|---|---|---|
|1980s|**MS-DOS**|Disk Operating System — pure command-line, three core files (`COMMAND.COM`, `IO.SYS`, `MSDOS.SYS`)|
|Early 1990s|**Windows 3.x**|Ran _on top of_ DOS; DOS Prompt still accessible inside Windows|
|Mid-1990s|**Windows NT**|Replaced the DOS kernel entirely with the NT kernel (still used today in Windows 10/11 and Server)|
|NT onwards|**Command Prompt**|An _emulated_ DOS environment (`cmd.exe`) to maintain backward compatibility|
|2006+|**PowerShell**|A modern, full-featured management shell that replaced the aging Command Prompt|

> The Windows Command Prompt (`cmd.exe`) is essentially an emulated DOS environment for backward compatibility. PowerShell is the modern successor — a true grown-up management shell.

### Why Use PowerShell Instead of the GUI?

- **Scalability** — Simultaneously manage _large numbers_ of Windows systems.
- **Headless servers** — Windows Server can be installed _without a GUI_ (called Server Core). PowerShell is then the only management interface.
- **Automation** — Automate repetitive tasks with scripts.
- **More power** — PowerShell exposes far more information and control than any GUI tool.

## 1.2 PowerShell Tools

### Windows PowerShell Console

The standard terminal. Launch it from the Start menu.

> **Run as Administrator:** Many management tasks require elevated privileges. Simply being _logged in_ as an admin is not enough. Right-click the PowerShell icon → **Run as Administrator**. The title bar will show **"Administrator: Windows PowerShell"** when elevated.

### Windows PowerShell ISE (Integrated Scripting Environment)

A graphical editor split into three panes:

- **Top pane** — Script editor with syntax highlighting and line numbers. Files are saved as `.ps1`.
- **Bottom pane** — Live PowerShell console.
- **Right pane** — Searchable list of all cmdlets with help and parameter insertion.

> The ISE has autocomplete — start typing a cmdlet and it shows suggestions.

### Windows Terminal (Windows 11+)

A modern tabbed terminal application. Each tab is an **independent PowerShell session** (variables in one tab are not visible in another).

- Tabs can be renamed and colour-coded.
- Supports PowerShell, Command Prompt, and Azure Cloud Shell in separate tabs.

### Notepad (or any text editor)

You can write PowerShell scripts in any text editor. The instructor uses **Notepad** throughout the course for simplicity.

## 1.3 The Cmdlet Syntax

### Verb-Noun Structure

Every PowerShell command (called a **cmdlet**, pronounced "command-let") follows this pattern:

```
Verb-Noun
```

- **Verb** = the action (Get, Set, Start, Stop, New, Remove…)
- **Noun** = the object to act on (Service, Process, VM, Command…)

```powershell
Get-Service      # Retrieve a list of system services
Stop-Service     # Stop a service
Start-Service    # Start a service
Get-Process      # Retrieve running processes
```

> **Why:** Because verbs and nouns are reused across cmdlets, you can _guess_ what a cmdlet might be called even before looking it up.

### Adding Parameters

After the cmdlet name, you can pass parameters:

```powershell
Get-Service -Name "VSS"      # Get a specific service by name
Get-Service VSS              # Name parameter is often positional
Stop-Service -Name "VSS"
```

### The Pipeline (`|`)

The pipe symbol passes the output of one cmdlet as input to the next:

```powershell
Get-Service | Select-Object Name, Status
Get-Process | Format-List
```

## 1.4 Discovering Cmdlets with `Get-Command`

```powershell
# Find all cmdlets that start with a specific verb
Get-Command Get-*

# Find all cmdlets that use a specific noun
Get-Command *-Service

# Find cmdlets using a specific verb
Get-Command Resume-*
```

> Microsoft reuses common verbs (`Get`, `Set`, `New`, `Remove`, `Start`, `Stop`) and common nouns (`Service`, `Process`, `Item`) heavily. Learning these patterns lets you discover cmdlets intuitively.

## 1.5 Getting Help

### `Get-Help`

```powershell
Get-Help Get-Service        # Show help for a cmdlet
Get-Help Get-Service -Full  # Full documentation including all parameters
Get-Help Get-Service -Examples  # Show practical usage examples
```

Help output shows:

- **SYNTAX** — All possible parameter combinations
- **PARAMETERS** — What each parameter does
- Parameters in `[brackets]` are optional
- Parameters without brackets are required

### Updating Help Files

```powershell
Update-Help    # Download the latest help content from Microsoft
```

## 1.6 Using `Select-Object`

`Select-Object` filters which properties are shown, or forces PowerShell to display properties it would otherwise hide.

```powershell
# Show only specific columns
Get-Service VSS | Select-Object Name, Status, ServiceType

# Show ALL available properties of an object
Get-Service VSS | Select-Object *

# Expand a nested property (shown in {brackets} by default)
Get-VM VM1 -ComputerName HyperV | Select-Object -ExpandProperty HardDrives
```

### Table vs List View

```powershell
Get-Service | Format-Table    # ft (alias)
Get-Service | Format-List     # fl (alias)
```

## 1.7 Predicting a Cmdlet's Behaviour

Use the `-WhatIf` parameter on any cmdlet to see what _would_ happen without actually doing it:

```powershell
Stop-Service VSS -WhatIf     # Shows what would happen, does nothing
```

Use `-Confirm` to be prompted before execution:

```powershell
Stop-Service VSS -Confirm
```

## 1.8 Dealing with "Term Not Recognized" Errors

This error means PowerShell doesn't know what command you typed.

**Common causes:**

1. **Typo** — Check your spelling carefully.
2. **Missing module** — The cmdlet exists but its module isn't loaded yet.
3. **Execution policy** — Scripts may be blocked (covered in Chapter 3).
4. **Using a DOS command** — Some old DOS commands don't work in PowerShell.

**Troubleshooting steps:**

```powershell
# 1. Check if the cmdlet exists
Get-Command *keyword*

# 2. Check what modules are loaded
Get-Module

# 3. See all available modules
Get-Module -ListAvailable

# 4. Import a specific module
Import-Module ModuleName
```
